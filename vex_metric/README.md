# VEX: Virtual Exam Evaluation Pipeline

This repository contains the evaluation pipeline for **VEX**, a virtual-exam-based evaluation protocol for automatic short-answer grading (ASAG).

The pipeline evaluates grading models at two complementary levels:

1. **Item level**  
   Standard ASAG evaluation over individual student answers.

2. **Exam level**  
   Virtual exams are constructed by sampling question sets and evaluating model predictions after aggregation over students and questions.

The goal is to make model evaluation closer to the downstream educational setting, where grading errors matter not only per item but also after aggregation into exam-level student outcomes.

---

## 1. Overview

The VEX pipeline performs three main steps:

1. Create a reproducible virtual test environment.
2. Convert the test environment into a joined evaluation dataframe.
3. Compute item-level and exam-level metrics for all configured models.

From the repository root, the full pipeline can be executed with:

```bash
python vex_metric/run_vex.py
```

This runs the following scripts in sequence:

```text
create_vex_test_env.py
create_dataframe.py
evaluate_dataframe.py
```

## 4. Configuration

The main configuration is located in:

```text
vex_config.py
```

Important settings:

```python
N_RUNS = 500
TEST_SIZES = [10, 15]
RANDOM_SEED = 4242
```

These settings mean that the pipeline creates:

- 500 sampled virtual tests,
- for each test size in `[10, 15]`,
- using a fixed random seed for reproducibility.

The output environment is written to:

```python
TEST_ENV_FOLDER = "vex_test_env"
```

The joined dataframe used for evaluation is written to:

```python
OUTPUT_PARQUET = Path(TEST_ENV_FOLDER) / TESTS_DATAFRAME / "dataframe_env.parquet"
```

## 5. Step 1: Create Virtual Test Environment

Script:

```bash
python create_vex_test_env.py
```

This script samples virtual exams from the input parquet.

For each run and each test size, it samples a set of question IDs without replacement. A student is eligible for a sampled virtual exam only if the student has a valid answer (if an answer_id exists for this question) for every sampled question.

For example, for a 10-question virtual exam, a student is included only if they answered all 10 sampled questions.

Generated files include:

```text
vex_test_env/
└── 3_tests/
    └── test_1/
        ├── questions/
        │   ├── questions_10.txt
        │   └── questions_15.txt
        ├── students/
        │   ├── students_10.txt
        │   └── students_15.txt
        └── metadata/
            ├── 1_metadata_questions_10.txt
            └── 1_metadata_questions_15.txt
```

Each `questions_*.txt` file contains the sampled question IDs.

Each `students_*.txt` file contains the eligible student IDs for that virtual exam.

Each metadata file contains information such as:

- sampled question IDs,
- eligible students,
- number of rows,
- number of valid answers,
- number of empty answer texts,
- topic distribution,
- Bloom-level distribution.

The script also writes global metadata to:

```text
vex_test_env/1_test_env_metadata/test_env_metadata.txt
```

If the environment already exists and matches the current configuration, it is reused. If the configuration or metadata changed, the environment is recreated.

## 8. Item-Level Evaluation

Item-level metrics are computed directly on the original held-out input parquet, not on the synthetic virtual-exam dataframe.

This is intentional.

The item-level source is:

```python
INPUT_PARQUET
```

The item-level evaluation computes:

| Metric | Meaning |
|---|---|
| `item_mae` | Mean absolute error |
| `item_mse` | Mean squared error |
| `item_rmse` | Root mean squared error |
| `item_qwk` | Quadratic weighted kappa |

The item-level evaluation is useful for comparison with standard ASAG work, but it does not capture how errors accumulate or cancel out at the student/exam level.

---

## 9. Exam-Level Evaluation

For every virtual exam, student-level totals are computed.

For each student:

```text
gold_total = sum of human grades over sampled questions
pred_total = sum of model grades over sampled questions
```

The normalized scores are:

```text
gold_norm = gold_total / test_size
pred_norm = pred_total / test_size
```

A student is retained for an exam-level metric only if they have:

- exactly `test_size` rows,
- exactly `test_size` unique questions,
- exactly `test_size` valid human grades,
- exactly `test_size` valid model predictions.

This prevents incomplete virtual exams from biasing the aggregated metrics.

---

## 10. Exam-Level Metrics

The exam-level evaluation computes the following metric groups.

### 10.1 Rank Consistency

```text
EL-tau_b
```

This is Kendall's tau-b between human and model student totals.

It measures whether the model preserves the ranking of students after aggregating over the virtual exam.

---

### 10.2 Absolute Linear Scale

The absolute linear scale maps normalized scores to an absolut grade scale:

```text
linear_grade = 1.0 + 5.0 * normalized_score
```

With the default configuration:

```python
LINEAR_MIN_GRADE = 1.0
LINEAR_MAX_GRADE = 6.0
LINEAR_ROUNDING_STEP = 0.5
LINEAR_PASS_THRESHOLD_NORM = 0.6
```

A normalized score of `0.6` corresponds to grade `4.0`, the pass threshold.

Metrics:

| Metric | Meaning |
|---|---|
| `EL-Acc Linear Abs` | Exact agreement of rounded linear grades |
| `EL-QWK Linear Abs` | QWK over rounded linear grades |
| `EL-PassAcc Linear Abs` | Pass/fail accuracy |
| `EL-PassQWK Linear Abs` | QWK over pass/fail decisions |

---

### 10.3 Mean-Centered Linear Scale

The mean-centered scale is cohort-dependent.

It maps scores below and above the cohort mean differently:

```text
if T <= mean:
    z = 0.5 * T / mean

if T > mean:
    z = 0.5 + 0.5 * (T - mean) / (max_total - mean)
```

The resulting normalized value `z` is then mapped to the same Swiss-style grade range.

This scale is included as a sensitivity analysis. It should not be interpreted as the main absolute grading scheme because it depends on the cohort distribution.

Metrics:

| Metric | Meaning |
|---|---|
| `EL-Acc Linear Mean` | Exact agreement of mean-centered rounded grades |
| `EL-QWK Linear Mean` | QWK over mean-centered rounded grades |
| `EL-PassAcc Linear Mean` | Pass/fail accuracy |
| `EL-PassQWK Linear Mean` | QWK over pass/fail decisions |

---

### 10.4 Bologna Scale

The Bologna scale assigns categorical labels:

```text
A, B, C, D, E, F
```

Students below the absolute pass threshold receive:

```text
F
```

Passing students are ranked by total points and assigned to passing categories using the configured distribution:

```python
BOLOGNA_PASSING_DISTRIBUTION = [0.10, 0.25, 0.30, 0.25, 0.10]
BOLOGNA_PASSING_LABELS = ["A", "B", "C", "D", "E"]
BOLOGNA_FAIL_LABEL = "F"
```

Ties are handled by assigning identical point totals to the same Bologna category. The category is chosen using the midpoint rank of the tied group.

Metrics:

| Metric | Meaning |
|---|---|
| `EL-Acc Bologna` | Exact Bologna label agreement |
| `EL-QWK Bologna` | QWK over ordered Bologna labels |

The ordered label sequence is:

```python
BOLOGNA_ORDERED_LABELS = ["F", "E", "D", "C", "B", "A"]
```

## 11. Report Format

The main report contains one section per test size and one section over all test sizes:

```text
VEX EVALUATION REPORT - TEST SIZE 10
VEX EVALUATION REPORT - TEST SIZE 15
VEX EVALUATION REPORT - ALL TEST SIZES
```

For each model, the report includes:

```text
Item-level MAE
Item-level MSE
Item-level RMSE
Item-level QWK

EL-tau_b

EL-Acc Linear Abs
EL-QWK Linear Abs
EL-PassAcc Linear Abs
EL-PassQWK Linear Abs

EL-Acc Linear Mean
EL-QWK Linear Mean
EL-PassAcc Linear Mean
EL-PassQWK Linear Mean

EL-Acc Bologna
EL-QWK Bologna
```

Exam-level metrics are reported as:

```text
mean ± standard deviation
```

The mean and standard deviation are computed over the sampled virtual exam instances.

For example:

```text
EL-tau_b: 0.8123 ± 0.0412
```

This means that the metric was first computed separately for each sampled virtual exam and then aggregated across virtual exams.

## 12. Reproducibility

The virtual exam sampling is deterministic given:

```python
RANDOM_SEED = 4242
N_RUNS = 500
TEST_SIZES = [10, 15]
INPUT_PARQUET = Path("dataset/additional/vex_metric_dataset/merged_model_predictions.parquet")
```

The global environment metadata is stored in:

```text
vex_test_env/1_test_env_metadata/test_env_metadata.txt
```

If this metadata matches the current configuration, the virtual test environment is reused.

If the metadata does not match, the environment is regenerated.

This ensures that reported results can be traced back to the exact sampled virtual exams.

---

## 13. Duplicate and Completeness Checks

The pipeline includes several hard validation checks.

It rejects:

- missing required columns,
- missing model prediction columns,
- duplicate `answer_id` values in the answer master table,
- duplicate student-question pairs in the item-level source,
- duplicate `(test_id, test_size, member_id, question_id)` pairs in the virtual exam dataframe.

This is important because duplicate student-question rows would inflate exam totals and invalidate exam-level results.

The environment creation step also writes a duplicate report:

```text
vex_test_env/1_test_env_metadata/duplicate_student_question_combinations.tsv
```

---

## 14. Missing Predictions

At item level, missing model predictions are counted and excluded from metric computation.

At exam level, a student is excluded from a specific model's exam-level metric if the student does not have a complete set of valid model predictions for all sampled questions.

The report includes:

```text
Exam students raw total
Exam students valid total
Students dropped incomplete
Students missing human
Students missing predictions
```

This makes it visible whether a model's exam-level result is based on fewer complete student exams.

## 17. Important Design Decisions

### Item-level metrics are computed on the original held-out input

The synthetic virtual-exam dataframe contains repeated use of the same held-out answers across many sampled exams. Therefore, item-level metrics are computed directly on the original held-out input parquet to avoid artificially inflated item counts.

### Exam-level metrics are computed on virtual exams

The virtual exams simulate downstream educational use: students receive aggregate outcomes based on multiple answers, not isolated item scores.

### Question sampling is question-based

Questions are sampled without replacement for each virtual exam. Eligible students are then selected based on whether they answered all sampled questions.

### Empty answer text is allowed

Rows with empty answer text are not automatically removed if they have valid identifiers. This is intentional because an empty answer can be a valid student response and should be graded accordingly.

### Missing identifiers are removed

Rows without valid `question_id`, `member_id`, or `answer_id` are removed during environment creation because they cannot support valid virtual exam construction.
